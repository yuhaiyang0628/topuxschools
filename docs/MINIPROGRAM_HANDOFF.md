# 微信小程序交接

## 已完成的结构

`miniprogram` 是可直接导入微信开发者工具的原生小程序工程，包含五个底部入口：

1. 首页：项目、案例、笔记的入口与精选内容。
2. 学校列表：地区、条件、关键词筛选；每次加载 20 个项目；项目详情与官网链接复制。
3. 案例分享：默认全世界；点击 US、CAN、UK、AUS、EU、Asia 进行筛选；再次点击同一地区取消；每次加载 6 个案例。
4. 干货笔记：文章流与独立阅读页。
5. 答疑解惑：低压力的联系入口，当前提供邮箱复制，等待补充微信号或二维码。

内容服务分两种模式：

```text
网页数据文件
  -> scripts/build-miniprogram-content.mjs
  -> 小程序离线内容 data/content.js
  -> 云数据库导入文件 seed/*.json

小程序页面
  -> services/content.js
  -> 本地内容（默认）或 content 云函数（云端模式）
```

## 你回来后需要提供的内容

优先级从高到低：

1. 小程序 AppID：用于把工程从临时预览切换为你的正式小程序。
2. 云开发环境 ID：用于启用可持续更新的内容数据库。
3. 对外联系信息：至少一个真实邮箱；最好再提供微信号或二维码图片。
4. 小程序名称、头像与一句简介：用于公众平台资料、分享卡片和搜索展示。
5. 是否愿意启用登录与收藏：第一版可以不做，先让内容公开浏览；若要做收藏或咨询表单，需要设计用户数据与隐私说明。

不要在聊天或代码库中提供云开发 SecretId、SecretKey、微信登录密钥等长期凭证。当前工程接入云开发只需要 AppID 和环境 ID；如日后必须使用敏感凭证，应通过本地私密配置或云端环境变量管理。

## 首次接入步骤

1. 安装并打开微信开发者工具，导入 `miniprogram` 文件夹。
2. 在微信公众平台创建或选择你的小程序，复制 AppID。
3. 将 `miniprogram/project.config.json` 中的 `appid` 替换为该 AppID。
4. 在微信开发者工具的“云开发”入口创建环境，记录生成的环境 ID。首次创建通常需要同意服务条款并完成权限授权；环境创建后会与当前小程序关联。[官方创建环境说明](https://docs.cloudbase.net/quick-start/create-env)
5. 将 `miniprogram/data/config.js` 中的 `contentMode` 改为 `cloud`，并填入 `cloudEnvId`。
6. 在开发者工具中部署 `miniprogram/cloudfunctions/content` 云函数。
7. 在云数据库创建三个集合：`programs`、`caseStudies`、`articles`。为每个集合的 `id` 字段建立唯一索引，再把 `miniprogram/seed` 下同名 JSON 文件导入各自集合。
8. 将数据库读写权限设为“仅云函数可写”；客户端统一通过已部署的 `content` 云函数读取。这是避免公开内容库被人恶意修改的关键一步。
9. 重新编译，确认首页统计显示 155 个项目、6 个案例、4 篇笔记。此后新内容会先写入云端，再刷新页面显示。

云开发环境有唯一的环境 ID，包含独立的数据库、存储和云函数资源；微信开发者工具可以直接创建并关联小程序环境。[官方小程序快速开始](https://docs.cloudbase.net/quick-start/frameworks/wechat-miniprogram)

## 内容更新流程

目前 `content/` 下的 `programs.js`、`case-studies.js`、`articles.js` 是 Web 与小程序共用的内容源。每次更新内容后：

1. 更新网页对应的数据文件。
2. 运行 `node scripts/build-miniprogram-content.mjs`。
3. 本地预览确认网页与小程序内容正确。
4. 云端模式下，使用按 `id` 更新的同步工具把新内容写入对应集合。首次导入可以使用控制台；不要把全量 JSON 反复直接导入，否则可能产生重复记录。

当案例和笔记更新频率稳定后，下一步应把内容源迁移到一个后台编辑界面或内容管理系统。届时网页和小程序都只读同一个云端接口，不再通过手工同步文件。

字段定义和案例投稿审核规则见 [CONTENT_SCHEMA.md](CONTENT_SCHEMA.md)。

## 已预留、但暂未连接的能力

- 云数据库与云函数：代码和种子数据已准备，等待环境 ID 与首次部署。
- 正式联系入口：等待真实邮箱、微信号或二维码。
- 收藏、浏览历史、投稿表单：需要决定是否让用户登录，以及准备隐私政策。
- 站外项目官网：小程序当前复制官网链接，而不是内嵌网页。正式打开站外页面或使用 web-view 时，需要按平台规则配置业务域名并逐个核验链接。
- 内容后台：建议等至少积累 20 个案例、20 篇笔记后再做，避免过早投入管理系统。
- 可重复的云端内容同步：首次导入已准备完成；安全的按 `id` 更新脚本等待你确认以后由谁维护内容、以及选择的云端账号/环境后再接入。

## 产品建议

1. 第一版不要急着做“AI 选校”。你的已验证项目数据和真实案例才是最有辨识度的内容资产；AI 应放在数据充足后的辅助问答，而不是替代选校判断。
2. 案例收集需要明确授权。建议每个投稿都记录是否允许公开、能否展示院校/成绩、是否匿名、是否允许用于宣传，避免后期反复撤稿。
3. 学校项目会过期。为每条项目保留 `lastVerified`，并在前端展示“最后核验”；之后可增加“待复核”队列，优先检查截止日期与语言要求。
4. 小程序的自然传播不等于自然转化。每个案例和笔记都应该有明确但低压力的下一步，例如“收藏”“提交自己的问题”“投稿案例”；不要一上来使用强咨询话术。
5. 最容易被忽略的是隐私与运营负担：用户提交背景、成绩、作品集链接时属于敏感个人信息。第一版应尽量只允许公开内容浏览和轻量联系，等隐私说明、处理流程与回复节奏明确后再开表单。
